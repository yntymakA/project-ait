1. Общая цель мобильного приложения

Нужно разработать Flutter-мобильное приложение для пользователей marketplace-платформы. Приложение должно выглядеть и ощущаться как реальный MVP стартапа, а не как учебный прототип. Пользователь через приложение должен иметь возможность полноценно пользоваться платформой: регистрироваться, создавать и редактировать объявления, искать товары/услуги, общаться с продавцами, добавлять в избранное, оплачивать продвижение, получать уведомления, управлять профилем и языком интерфейса.

2. Технологические требования к мобильной части

Мобильное приложение обязательно должно быть написано на Flutter + Dart. Архитектура должна быть API-driven, то есть приложение работает как клиент, который получает и отправляет данные через API. Управление состоянием должно быть структурированным: допустимы Riverpod, Bloc/Cubit, Provider или другой внятно обоснованный вариант. Нельзя делать приложение как набор хаотичных setState без архитектуры.

3. Что должен уметь пользователь в приложении

Через приложение пользователь должен иметь возможность:

зарегистрироваться и войти в аккаунт;
просматривать и редактировать профиль;
создавать, редактировать, архивировать и управлять своими объявлениями;
загружать и управлять медиа для объявлений;
просматривать ленту объявлений;
выполнять поиск, фильтрацию, сортировку;
открывать карточку продавца;
видеть все объявления конкретного продавца;
добавлять объявления в избранное;
писать сообщения другим пользователям;
отправлять вложения в чатах;
жаловаться на объявления или пользователей;
менять язык интерфейса;
оплачивать функции платформы;
покупать продвижение/буст/таргетинг для объявлений;
получать и просматривать уведомления;
управлять своим аккаунтом и настройками.

Приложение должно корректно работать как минимум для двух пользовательских состояний:

гость;

Гость может:

смотреть детали объявления;
смотреть публичный профиль продавца.

Авторизованный пользователь дополнительно может:

хаотичные отступы;

В каждом объявлении должно быть ясно видно, кто продавец. Пользователь должен иметь возможность нажать на продавца и открыть его публичную страницу, где должны отображаться:

Также обязательно нужен отдельный экран со всеми объявлениями конкретного продавца с поддержкой:

11. Категории и динамические атрибуты

Приложение должно поддерживать категории объявлений. Минимально категория содержит:

id;
name;
slug/code;
is_active;
display_order;
created_at.

Желательно поддержать подкатегории через parent_category_id.

Для production-style подхода должны быть предусмотрены category-specific attributes, например:

для машин: brand, model, year, mileage, fuel type;
для недвижимости: rooms, area, floor, building type;
для электроники: brand, condition, warranty.

На уровне приложения это должно выражаться в том, что форма создания/редактирования объявления умеет показывать дополнительные поля в зависимости от выбранной категории.

16. Мессенджер

Мессенджер — обязательная часть приложения. Пользователь должен иметь возможность открыть объявление, нажать “написать продавцу”, после чего должна создаваться или переоткрываться беседа по контексту этого объявления.

Обязательные возможности чата
список диалогов;
экран конкретного диалога;
отправка текстового сообщения;
получение и отображение сообщений;
правильное отображение sender/receiver;
timestamps;
unread/read state;
preview последнего сообщения в inbox.
Бизнес-ограничения, которые должны быть видны в UX
нельзя писать самому себе;
ограниченные/заблокированные пользователи могут быть ограничены;
сообщения существуют внутри conversation;
conversation может быть привязана к объявлению.
17. Вложения в сообщениях

Вложения в чате обязательны. Минимально должны поддерживаться:

Пользователь должен иметь возможность:

просматривать превью, где это возможно;

На уровне мобильного UX желательно показать:

тип вложения;
ошибки неподдерживаемого формата/слишком большого файла.

На клиенте должны корректно отображаться:

cancelled;
